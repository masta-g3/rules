long-execute workflow variant

A bounded auto-continuation wrapper for approved plans: execute stays authoritative, while a small runtime controller decides whether another visible turn should continue the same work.

Why it exists

Some plans need more than one safe implementation turn.

Today, the maintainer repeatedly invokes execute for long plans. The proposal keeps each turn visible but removes the manual nudge when the agent explicitly says safe work remains.

Current

Manual continuation between slices of the same approved plan.

Proposed

Bounded continuation only after an exact final-line marker, with the same execute rules still governing the work.

Layer boundary

Semantics stay portable; continuation stays isolated.

The important boundary: skills/execute remains the source of truth; the new pieces only add a controlled way to ask for the next turn.

Continuation loop

The injected prompt is a normal follow-up user message, so the automation is visible in the conversation instead of hidden in a system prompt mutation.

Decision anchors

What this design buys, and what it refuses.

  • Contained drift: implementation policy remains in execute; long-execute only adds continuation labels.
  • Visible automation: continuation happens through a follow-up user message, not by appending invisible instructions.
  • Completion pressure: the prompts require an audit before READY FOR REVIEW, reducing shortcut “done” claims.
  • Scoped availability: pi/skills/ overlays to the Pi skill root only; shared harness skills stay unchanged.
  • No workflow creep: it does not auto-run review, reflect, or commit.
